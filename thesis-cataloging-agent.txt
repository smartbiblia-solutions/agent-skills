Your task is to generate a complete GitAgent-compliant, Nanobot/OpenClaw-compatible repository for a **self-improving French UNIMARC cataloguing agent**.

# Goal

Build an agent for AgentDesk, a local AI agent orchestration dashboard powered by **Nanobot (HKUDS)** as the sole execution engine.

The agent takes a thesis or dissertation cover page, extracts cataloguing metadata, checks Sudoc duplicates, aligns people and institutions with IdRef where possible, generates a UNIMARC XML bibliographic record, and improves its documentation from cataloguer feedback.

# Global Language Constraint

The agent MUST:

- understand French;
- respond in French;
- generate all natural-language outputs in French;
- keep structured outputs such as JSON, XML, ISO 2709, and config files in their required technical formats.

# Supported Cataloguing Scope

The agent supports ONLY two local library profiles:

1. `these_originale_imprimee`
   - Printed original doctoral thesis.
   - Excludes electronic theses, thesis reproductions, commercial editions, exercise theses, HDR, manuscripts, and uncertain cases.

2. `memoire_original_imprime`
   - Printed original dissertation / memoire.
   - Excludes electronic submissions, reproductions, HDR, commercial editions, and uncertain cases.

If the document is outside these two profiles, or if the profile cannot be inferred with high confidence, the agent MUST stop and return a French message requesting review by a human cataloguer.

# Critical Architecture

The scaffold MUST strictly separate four systems:

1. `/dataset/`
   - Offline ground truth: image or OCR text -> extracted JSON -> expected UNIMARC.
   - Used only for evaluation, regression tests, or future fine-tuning.
   - NEVER modified during normal agent execution.

2. `/raw/`
   - Immutable source documentation.
   - Contains scraped or manually imported ABES/Sudoc/UNIMARC source pages and library specifications.
   - The agent MUST NOT edit files in `/raw/`.

3. `/knowledge/`
   - Maintained LLM Wiki in French.
   - Contains operational cataloguing rules, field mappings, examples, edge cases, and decisions extracted from `/raw/` and human feedback.
   - This is the source of truth during cataloguing runs.

4. `/memory/`
   - Experience memory.
   - Stores past cases, corrections, validation failures, and feedback-derived lessons.
   - Used at inference time but does not modify model weights.

Optional derived index:

5. `/graphify-out/`
   - Optional generated knowledge graph artifacts from Graphify.
   - Derived from `/raw/` and/or `/knowledge/`.
   - NEVER treated as source of truth.

NEVER merge these systems.

# Sudoc Duplicate Check

The agent MUST verify that no Sudoc record already exists before generating a new printed record.

External skill:

- `search-records-sudoc`

The duplicate check has two ordered phases.

## Phase 1 - Printed Record Detection

First, search strictly for an existing printed thesis or memoire record.

Search constraints:

- include only printed theses and memoires;
- exclude electronic records, reproductions, and non-supported profiles where possible;
- identify and return the Sudoc PPN for any probable match.

Required query sequence for printed records:

1. strict: title + author + defense year;
2. relaxed: title + author;
3. fallback: title only.

If a probable printed match is found, STOP and return:

```text
Une notice existe déjà dans le Sudoc. PPN : {ppn}
```

## Phase 2 - Electronic Version Detection

If no printed record is found, search for an existing electronic record for the same document.

Search constraints:

- include only electronic theses or memoires;
- match the same intellectual work, not merely a similar title;
- identify and return the Sudoc PPN for any probable electronic match.

Required query sequence for electronic records:

1. strict: title + author + defense year;
2. relaxed: title + author;
3. fallback: title only.

If a probable electronic match is found, STOP and return:

```text
Une notice existe déjà pour la version électronique dans le Sudoc. PPN : {ppn}
```

If no printed or electronic match is found, continue the cataloguing pipeline.

# IdRef Alignment

External skill:

- `search-authorities-idref`

Internal skill:

- `enrich-with-idref`

For each person or institution, the agent MUST:

1. retrieve candidates from IdRef;
2. compare normalized name, discipline, institutional context, and known works;
3. assign a PPN only for a strong match;
4. explain ambiguous candidates;
5. return `aucune autorité IdRef pertinente trouvée` when no reliable match exists.

Forbidden:

- hallucinating PPNs;
- forcing matches;
- matching the uncatalogued document by title as if it already existed.

PPNs MUST be injected in UNIMARC authority subfield `$3` where relevant, especially `700`, `701`, `702`, and `711`.

# UNIMARC Generation Rules

The UNIMARC record MUST be generated from validated JSON and grounded in `/knowledge/`.

The scaffold MUST create knowledge files that specify indicators explicitly. At minimum, examples MUST cover:

- `181 ##$P01$ctxt`
- `182 ##$P01$cn`
- `183 ##$P01$anga`
- `328 #0$b{degree}$c{discipline}$e{institution}$d{year}`
- `686 ##$a{code}$2TEF`
- `608 ##$3027253139$aThèses et écrits académiques$2rameau`

Profile-specific fields:

## `these_originale_imprimee`

- `008`: generated according to the local library profile for printed original theses.
- `029 ##$aFR`
- `102 ##$aFR`
- `104 ##$ak$by$cy$dba$e0$ffre`
- `105`: use the exact local library specialist rule from `/knowledge/unimarc/profiles.md`; do NOT infer from memory. In particular, do NOT use `$bv` merely because the document is a memoire.
- `106 ##$ar`
- `181 ##$P01$ctxt`
- `182 ##$P01$cn`
- `183 ##$P01$anga`
- `328 #0` thesis note is mandatory.
- `686 ##$a{discipline.code}$2TEF` is mandatory for original doctoral theses, except unsupported exercise-thesis cases which must stop for review.

## `memoire_original_imprime`

- Use the same printed text carrier/content rules unless `/knowledge/` states a local exception.
- `328 #0` dissertation/memoire note is mandatory.
- `686 ##$a{discipline.code}$2TEF` is NOT automatic; generate it only if the local library profile explicitly requires it.

# JSON to UNIMARC Mapping

The scaffold MUST encode this mapping in `/knowledge/unimarc/json-to-unimarc.md` and in the generation skill:

- `title` -> `200 $a`
- `subtitle` -> `200 $e`
- `author` -> `200 $f` and `700 $4070`
- `advisor` -> `200 $g` preceded by `sous la direction de`; also `701 $4727` for thesis advisors or `701 $4003` for memoire advisors, according to the profile rule
- `defense_year` -> `100 $a`, `214 $d`, `328 $d`
- `abstract` -> `330 $a`
- `jury_president` -> `701 $4956`
- `reviewers` -> `701 $4958`
- `committee_members` -> `701 $4555`
- `granting_institution` -> `711 $4295`
- `co_tutelle_institutions` -> `711 $4995`
- `doctoral_school` -> `711 $4996`
- `partner_institutions` -> `711 $4985`
- `discipline.code` -> `686 $a` when the active profile requires `686`

# UNIMARC Format Conversion

External skill:

- `convert-records-unimarc`

This is optional export functionality only. It MUST NOT modify bibliographic content or cataloguing decisions.

Supported formats:

- UNIMARC/XML;
- UNIMARC/JSON using pymarc dictionary structure;
- ISO 2709 binary.

Conversion strategy:

```text
XML -> JSON -> ISO
ISO -> JSON -> XML
JSON <-> XML
JSON <-> ISO
```

The pivot format MUST be JSON. Leader length MUST be normalized to 24 characters before parsing.

# Self-Improvement

Triggers:

- user correction;
- validation failure;
- low confidence;
- unresolved ambiguity.

Process:

1. compare generated output with corrected output;
2. identify the error type;
3. extract a general rule;
4. store the case in `/memory/`;
5. propose a wiki update in French;
6. update `/knowledge/` only when the rule is verified or human-approved.

Example rules:

- `Si plusieurs directeurs sont présents, répéter la zone 701 avec le code de fonction approprié.`
- `Si plusieurs établissements de cotutelle sont présents, répéter la zone 711 avec $4995.`
